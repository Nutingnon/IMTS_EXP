# 数据处理声明

## 1. 说明范围

本文档说明 APN 项目中不规则时序数据的处理链路，覆盖以下四个数据集实现：

1. P12: data/data_provider/datasets/P12.py
2. HumanActivity: data/data_provider/datasets/HumanActivity.py
3. MIMIC_III: data/data_provider/datasets/MIMIC_III.py (MIMIC_III 不能完全自动下载，先要把 complete_tensor.csv 放到 ~/.tsdm/rawdata/MIMIC_III_DeBrouwer2019/，再运行 MIMIC_III.sh。)
4. USHCN: data/data_provider/datasets/USHCN.py

## 2. 公共加载框架

1. 数据集模块按 `dataset_name` 动态导入，统一入口为 `data/data_provider/data_factory.py`。
2. `collate_fn` 也按配置名动态绑定，来自对应数据集模块。
3. 配置来源以命令行参数为准，不由 yaml 反向覆盖（见 `utils/configs.py` 注释）。

## 3. P12 当前结论

### 3.1 数据来源与任务入口

1. Data 包装入口在 `data/data_provider/datasets/P12.py`，底层调用 `data/dependencies/tsdm/tasks/P12.py` 的 `Physionet2012` 任务。
2. tsdm 默认使用用户家目录下 `.tsdm` 作为根目录（配置见 `data/dependencies/tsdm/config/config_files/config.yaml`）。
3. P12 原始数据位于 `.tsdm/rawdata/Physionet2012`，处理后数据位于 `.tsdm/datasets/Physionet2012`。

### 3.2 切分逻辑

1. 任务层按 ID 做 train/test，再从 train 中切出 val。
2. Data 层按 `flag in {train, val, test, test_all}` 调用对应 split。

### 3.3 三种 collate

1. `collate_fn`: 常规不规则对齐与 padding。
2. `collate_fn_patch`: 先按时间窗口分 patch，再在 patch 级别 padding，最后 reshape 回统一长度。
3. `collate_fn_tpatch`: 更细粒度 patch 变体（按变量展开的 patch 处理）。

## 4. HumanActivity 数据处理逻辑

### 4.1 数据来源与落盘

1. Data 类通过 `HumanActivity(root=dataset_root_path)` 加载数据。
2. HumanActivity 依赖在 root 下维护：
   - `raw/`
   - `processed/`
3. 处理后文件保存为 `processed/data.pt`。

### 4.2 切分与样本构造

1. 先做 seen/test = 90/10（`shuffle=False`）。
2. 再对 seen 做 train/val = 90/10（`shuffle=False`）。
3. train/val/test 均经过 `Activity_time_chunk` 构造监督样本。

### 4.3 collate 逻辑

1. `collate_fn`
2. `collate_fn_patch`
3. `collate_fn_tpatch`

三者命名和 P12 一致，整体处理风格一致。

### 4.4 约束

1. `seq_len + pred_len <= 4000`。

## 5. MIMIC_III 数据处理逻辑

### 5.1 数据来源与落盘

1. Data 类调用 tsdm 任务 `MIMIC_III_DeBrouwer2019`。
2. 原始文件要求为 `complete_tensor.csv`。
3. 该文件需放入 tsdm 的 `RAWDATA_DIR`（默认位于 `~/.tsdm/rawdata/MIMIC_III_DeBrouwer2019`）。

### 5.2 切分与样本构造

1. 任务层按 ID 做 train/test，再从 train 中切出 val（顺序切分，`shuffle=False`）。
2. Data 层根据 flag 调用 `task.get_dataset((0, flag))`，`test_all` 时合并 train+val+test。

### 5.3 collate 逻辑

1. `collate_fn`
2. `collate_fn_patch`
3. `collate_fn_tpatch`

### 5.4 约束

1. `seq_len + pred_len <= 96`。

## 6. USHCN 数据处理逻辑

### 6.1 数据来源与落盘

1. Data 类调用 tsdm 任务 `USHCN_DeBrouwer2019`。
2. 原始文件名为 `small_chunked_sporadic.csv`。
3. 落盘目录遵循 tsdm 配置：
   - 原始数据目录：`~/.tsdm/rawdata/USHCN_DeBrouwer2019`
   - 处理后目录：`~/.tsdm/datasets/USHCN_DeBrouwer2019`

### 6.2 切分与样本构造

1. 任务层按 ID 做 train/test，再从 train 中切出 val。
2. Data 层根据 flag 获取对应 split，`test_all` 时合并三份数据。

### 6.3 collate 逻辑

1. `collate_fn`
2. `collate_fn_patch`
3. `collate_fn_tpatch`

### 6.4 约束

1. `seq_len + pred_len <= 200`。

## 7. 四个数据集的一致性与差异总结

### 7.1 一致性

1. 都使用 Data 封装 + flag 切分接口。
2. 都提供 `collate_fn / collate_fn_patch / collate_fn_tpatch` 三种批处理。
3. 都在 preprocess 中统计 `seq_len_max_irr` 与 `pred_len_max_irr`；patch 模式额外统计 `patch_len_max_irr`。

### 7.2 差异

1. HumanActivity 使用项目内自定义依赖，直接在 `dataset_root_path` 下构建 `raw/processed`。
2. P12、MIMIC_III、USHCN 使用 tsdm 数据管理与任务切分逻辑。
3. 不同数据集最大时间范围不同，patch 边界归一化分母也不同：
   - P12: 48
   - MIMIC_III: 96
   - USHCN: 200
   - HumanActivity: 使用 `seq_len + pred_len` 作为归一化总长度。

## 8. 备注

1. 同一数据集下，不同模型脚本是否“处理完全一致”，关键看是否显式传入 `--collate_fn collate_fn_patch` 等参数。
2. 若脚本未指定 `--collate_fn`，默认使用 `collate_fn`。
